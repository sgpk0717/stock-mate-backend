## 1. 시뮬레이션이란?

### 왜 시뮬레이션이 필요한가?

백테스트는 **과거 데이터**로 전략을 검증합니다. 하지만 과거에 일어나지 않은 극한 상황 — 플래시 크래시, 유동성 위기, 금리 급등 — 에서 전략이 어떻게 작동할지는 알 수 없습니다.

**ABM(Agent-Based Model) 시뮬레이션**은 이 한계를 극복합니다:

- **가상 시장**을 만들어 수백 명의 다양한 투자자(에이전트)가 거래합니다
- **호가창(LOB)**이 실제처럼 작동합니다 — 지정가/시장가 주문, 가격-시간 우선 매칭
- 원하는 시점에 **인위적 충격 시나리오**를 주입합니다
- 사용자의 전략을 이 가상 시장에 넣어 **전략이 살아남는지** 확인합니다

### MCP Data Bus란?

**MCP(Model Context Protocol)**는 AI 에이전트가 외부 시스템과 소통하는 표준 프로토콜입니다.

Stock Mate의 MCP 서버는:
- AI가 **주식 데이터 조회, 뉴스 감성 확인, 주문 실행**을 할 수 있는 도구 5개를 제공합니다
- **거버넌스 규칙**으로 AI의 행동을 제한합니다 (주문 수량 한도, 실거래 차단 등)
- 모든 도구 호출을 **감사 로그**로 기록합니다

## 2. 시뮬레이션 페이지 진입하기

### 화면 접근 경로

1. 브라우저에서 Stock Mate 앱에 접속합니다 (`http://localhost:5173`)
2. **왼쪽 사이드바**에서 **"시뮬레이션"** 메뉴를 클릭합니다
3. 시뮬레이션 페이지가 열리며, 상단에 **2개 탭**이 표시됩니다

### 2개 탭 구조

| 탭 | 이름 | 용도 | 난이도 |
|----|------|------|--------|
| 1 | **스트레스 테스트** | ABM 가상 시장에서 시나리오 충격 테스트 | 초급~중급 |
| 2 | **MCP 대시보드** | AI 도구 서버 상태, 거버넌스, 감사 로그 관리 | 중급~고급 |

처음 사용하는 분은 **"스트레스 테스트" 탭**부터 시작하세요.

## 3. 스트레스 테스트 탭 — 전체 흐름

### 화면 구성

스트레스 테스트 탭은 **좌측(설정 + 결과)** + **우측(실행 기록)** 2열 레이아웃입니다.

```
┌─────────────────────────────────┬──────────────┐
│ [스트레스 테스트 설정]          │  실행 기록    │
│  - 테스트 이름                  │              │
│  - 시나리오 선택 (4종)          │  ┌──────────┐│
│  - 커스텀 시나리오 (AI)         │  │ 금리 충격 ││
│  - 시나리오 주입 시점           │  │ 완료 0.2% ││
│  - [+ 고급 설정]               │  ├──────────┤│
│  - [스트레스 테스트 시작]       │  │ 유동성    ││
│                                 │  │ 실행중 42%││
│ [진행률 바]  ████████░░ 80%    │  ├──────────┤│
│                                 │  │ 플래시    ││
│ [지표 카드 × 8]                │  │ 완료 0.3% ││
│ [가격 추이 차트]               │  └──────────┘│
│ [호가 깊이 시각화]             │              │
└─────────────────────────────────┴──────────────┘
```

### 전체 실행 흐름

```
시나리오 선택 → 설정 조정(선택) → "시작" 클릭
       │
       ▼
[서버: 에이전트 150명 생성 + 호가창 초기화]
       │
       ▼
[500스텝 시뮬레이션 실행] ← WebSocket 진행률 실시간 전송
       │
  스텝 250 (중간): 시나리오 충격 주입
       │
       ▼
[결과 저장 + DB 기록]
       │
       ▼
지표 카드 + 가격 차트 + 호가 깊이 표시
```

## 4. 시나리오 선택

### 프리셋 시나리오 4종

시나리오 선택 영역에 **4개 카드**가 2×2 그리드로 표시됩니다. 원하는 시나리오를 클릭하면 파란색 테두리로 선택됩니다.

| # | 시나리오 | 한글명 | 실제 효과 |
|---|---------|--------|----------|
| 1 | `rate_shock` | **금리 충격** | 펀더멘탈 에이전트의 내재가치 -5%, 유동성 30% 소멸 |
| 2 | `liquidity_crisis` | **유동성 위기** | 노이즈 트레이더 80% 탈출, 스프레드 5배 확대 |
| 3 | `flash_crash` | **플래시 크래시** | 대규모 시장가 매도 주문 투입, 순간 폭락 |
| 4 | `supply_chain` | **공급망 충격** | 내재가치 -15%, 모멘텀 추종 1.5배 강화 |

### 각 시나리오 상세 설명

#### 금리 충격 (Rate Shock)

현실 예시: 한국은행이 기준금리를 50bp 인상 발표

**시뮬레이션 효과:**
- 펀더멘탈 에이전트의 내재가치 목표가 **5% 하향** 조정됩니다
- 이들이 기존 보유분을 매도하기 시작합니다
- 전체 호가 유동성이 **30% 감소**합니다

**관전 포인트:** 가치 투자자들이 일제히 매도할 때 모멘텀 추종자들이 연쇄 매도하는지?

#### 유동성 위기 (Liquidity Crisis)

현실 예시: 증거금 콜 사태, 마진 청산 러시

**시뮬레이션 효과:**
- 노이즈 트레이더의 **80%가 시장을 이탈**합니다 (활동 중지)
- 매수-매도 스프레드가 **5배** 확대됩니다
- 호가창이 텅 비어 소량 주문에도 가격이 크게 움직입니다

**관전 포인트:** 유동성이 증발했을 때 전략이 빠져나올 수 있는지?

#### 플래시 크래시 (Flash Crash)

현실 예시: 2010년 미국 플래시 크래시, 알고리즘 매도 폭주

**시뮬레이션 효과:**
- 대량의 **시장가 매도 주문**이 한꺼번에 투입됩니다
- 호가창의 매수 잔량을 순식간에 소화시킵니다
- 가격이 급격히 하락한 뒤, `recovery_speed` 비율로 서서히 회복을 시도합니다

**관전 포인트:** 순간 폭락에서 전략의 손절이 제대로 작동하는지?

#### 공급망 충격 (Supply Chain)

현실 예시: 글로벌 반도체 수출 규제, 코로나 공급망 마비

**시뮬레이션 효과:**
- 펀더멘탈 내재가치가 **15% 급락**합니다 (산업 전체 충격)
- 차티스트 에이전트의 모멘텀 추종이 **1.5배 강화**됩니다 (하락 추세 가속)
- 노이즈 트레이더는 공포에 따라 매도 편향이 커집니다

**관전 포인트:** 펀더멘탈 훼손 + 패닉 매도 동시 발생 시 전략의 하방 리스크는?

### 시나리오 주입 시점

시나리오가 주입되는 스텝을 지정합니다. 기본값은 `250` (총 500스텝의 중간).

| 주입 시점 | 의미 | 용도 |
|----------|------|------|
| **50** | 시장 초기 (아직 안정화 전) | 변동성 높은 상태에서의 충격 반응 |
| **250** (기본) | 시장 중반 (안정 균형 상태) | 일반적인 충격 테스트 |
| **400** | 시장 후반 (에이전트 자산 변동 누적 후) | 피로 누적 상태에서의 충격 반응 |

## 5. 커스텀 시나리오 (AI 생성)

프리셋 4종으로 부족하다면, **자연어로 시나리오를 설명**하면 AI(Claude)가 시나리오 파라미터를 자동 생성합니다.

### 5.1 사용 방법

1. "커스텀 시나리오 (AI 생성)" 입력란에 시나리오를 설명합니다
2. **"생성"** 버튼을 클릭합니다
3. AI가 적절한 시나리오 타입과 파라미터를 결정합니다
4. 아래에 AI의 설명 텍스트가 표시됩니다
5. 자동으로 시나리오가 설정됩니다 (프리셋 선택은 해제됨)

### 5.2 입력 팁

**좋은 예시:**
- "반도체 수출 규제 발표로 산업 전체 충격" → supply_chain 시나리오 생성
- "대형 기관투자자 5000억 긴급 처분" → flash_crash 시나리오 생성
- "미국 연준 긴급 금리 인상 100bp" → rate_shock 시나리오 생성 (강화된 파라미터)
- "코로나급 시장 패닉, 유동성 동결" → liquidity_crisis 시나리오 생성

**나쁜 예시:**
- "시장 좀 안 좋게 해줘" (너무 추상적)
- "1초 만에 50% 폭락" (비현실적 → AI가 적절히 조정하지만 부정확할 수 있음)

### 5.3 AI 설명 읽기

생성 후 입력란 아래에 **AI의 설명 텍스트**가 표시됩니다:

> "반도체 수출 규제는 supply_chain 시나리오에 해당합니다. 내재가치를 -20% 하향하고
> 모멘텀 추종 강도를 2.0배로 설정했습니다. 이는 실제 2019년 일본 수출 규제 당시의
> 코스피 반도체 섹터 하락폭을 참고한 파라미터입니다."

## 6. 고급 설정 — 에이전트 & 거래소

**"+ 고급 설정"**을 클릭하면 에이전트 구성과 거래소 설정을 커스터마이즈할 수 있습니다.

### 6.1 에이전트 구성

| 항목 | 설명 | 기본값 | 범위 | 초보 권장 |
|------|------|--------|------|----------|
| **펀더멘탈** | 내재가치 기반 가치 투자자 수 | 20 | 0~100 | 20 |
| **차티스트** | 모멘텀/추세 추종 트레이더 수 | 30 | 0~100 | 30 |
| **노이즈** | 랜덤 매매 유동성 공급자 수 | 100 | 0~500 | 100 |
| **LLM** | Claude AI 기반 에이전트 수 | 0 | 0~20 | 0 (API 비용 발생) |

**에이전트 총 수 = 펀더멘탈 + 차티스트 + 노이즈 + LLM + StrategyAgent(1)**

기본 구성(총 151명)은 다양한 투자 성향이 섞인 현실적인 시장을 만듭니다.

### 6.2 에이전트별 역할 요약

```
[시장 참여자 구성]

┌── 펀더멘탈 (20명) ──────────────────────────────┐
│ "내재가치를 계산하고, 싸면 사고 비싸면 판다"      │
│ 안정화 역할: 가격이 펀더멘탈에서 벗어나면 복귀 유도│
└──────────────────────────────────────────────────┘

┌── 차티스트 (30명) ──────────────────────────────┐
│ "추세를 따른다. 오르면 사고, 내리면 판다"         │
│ 변동성 확대 역할: 상승/하락 추세를 가속             │
└──────────────────────────────────────────────────┘

┌── 노이즈 (100명) ──────────────────────────────┐
│ "무작위로 매수/매도 주문을 내놓는다"              │
│ 유동성 공급 역할: 호가창을 채워 거래가 이루어지게   │
└──────────────────────────────────────────────────┘

┌── LLM (0~20명) ─────────────────────────────────┐
│ "Claude가 직접 시장 상황을 분석하고 매매 결정"    │
│ 행동편향 내장: 손실회피(×2.5), 군집행동(0.3)      │
│ ⚠️ 20스텝마다 Claude API 호출 (비용 발생)        │
└──────────────────────────────────────────────────┘

┌── 전략 에이전트 (1명, 자동 생성) ───────────────┐
│ "사용자의 백테스트 전략을 가상 시장에서 실행"     │
│ strategy_json으로 전달된 매수/매도 조건 적용       │
└──────────────────────────────────────────────────┘
```

### 6.3 거래소 설정

| 항목 | 설명 | 기본값 | 범위 | 초보 권장 |
|------|------|--------|------|----------|
| **초기 가격** | 시뮬레이션 시작 시 주가 | 50,000 | 1 이상 | 50,000 |
| **틱 사이즈** | 호가 단위 (가격 변동 최소 단위) | 10 | 1 이상 | 10 |
| **총 스텝** | 시뮬레이션 전체 스텝 수 | 500 | 100~10,000 | 500 |
| **시드** | 랜덤 시드 (재현성 보장) | 미설정 | 정수 | 미설정 |

**시드(Seed)란?**
- 미설정: 매 실행마다 다른 결과 (랜덤)
- 숫자 입력 (예: 42): 같은 시드 → 동일한 결과 재현
- 두 시나리오를 비교할 때 같은 시드를 사용하면 **시나리오 변수만** 달라지므로 정확한 비교가 가능합니다

## 7. 테스트 실행 & 진행률

### 7.1 실행 시작

1. 시나리오를 선택합니다 (프리셋 또는 커스텀)
2. (선택) 테스트 이름을 입력합니다 — 비워두면 `"flash_crash 테스트"` 같은 기본명이 사용됩니다
3. **"스트레스 테스트 시작"** 버튼을 클릭합니다
4. 버튼이 **"실행 중..."**으로 바뀌며 비활성화됩니다

### 7.2 진행률 바

실행이 시작되면 설정 카드 아래에 **진행률 바**가 나타납니다:

```
┌─────────────────────────────────────────────┐
│ 시뮬레이션 실행 중...                  80%  │
│ ████████████████████████████████░░░░░░░░░  │
└─────────────────────────────────────────────┘
```

- **WebSocket** 연결로 실시간 업데이트됩니다
- 500스텝 기준 약 **5~15초** 소요됩니다 (LLM 에이전트가 있으면 더 오래 걸립니다)
- 완료 시 자동으로 결과가 표시됩니다

### 7.3 실패 시

시뮬레이션이 실패하면 **빨간색 에러 메시지**가 표시됩니다:

```
┌─────────────────────────────────────────────┐
│ ❌ ANTHROPIC_API_KEY not configured         │
└─────────────────────────────────────────────┘
```

흔한 실패 원인은 [트러블슈팅](#22-트러블슈팅) 섹션을 참고하세요.

## 8. 결과 분석 — 지표 카드 읽기

시뮬레이션 완료 후 **8개 지표 카드**가 2행 × 4열로 표시됩니다.

### 1행: 핵심 지표

```
┌────────────┐ ┌────────────┐ ┌────────────┐ ┌────────────┐
│ 전략 손익   │ │ MDD        │ │ 충격 깊이   │ │ 회복 시간   │
│ +0.15%     │ │ -0.52%     │ │ 0.33%      │ │ 0 스텝     │
└────────────┘ └────────────┘ └────────────┘ └────────────┘
```

| 지표 | 의미 | 좋은 값 |
|------|------|---------|
| **전략 손익** | StrategyAgent의 최종 PnL (%) | 양수 (빨간색 = 수익) |
| **MDD (최대 낙폭)** | 시뮬레이션 전 구간에서 최대 고점 대비 하락폭 | 0에 가까울수록 안정 |
| **충격 깊이** | 시나리오 주입 후 가장 깊은 하락폭 | 클수록 충격이 심함 |
| **회복 시간** | 충격 후 이전 고점의 99%까지 회복하는 데 걸린 스텝 수 | 0 = 미회복 또는 미하락 |

### 2행: 시장 상태 지표

```
┌────────────┐ ┌────────────┐ ┌────────────┐ ┌────────────┐
│ 최종 가격   │ │ 가격 변화   │ │ 변동성      │ │ 평균 스프레드│
│ 50,145     │ │ +0.29%     │ │ 4.73%      │ │ 30.0       │
└────────────┘ └────────────┘ └────────────┘ └────────────┘
```

| 지표 | 의미 | 해석 |
|------|------|------|
| **최종 가격** | 시뮬레이션 종료 시 가격 | 초기 50,000 대비 변동 확인 |
| **가격 변화** | (최종 - 초기) / 초기 × 100 | 시장 전체 방향성 |
| **연환산 변동성** | 일별 수익률 표준편차 × √252 × 100 | 15~25% = 일반적, 40%+ = 극단적 |
| **평균 스프레드** | 시뮬레이션 전 구간의 매수-매도 스프레드 평균 | 낮을수록 유동성 풍부 |

### 색상 의미

| 색상 | 의미 |
|------|------|
| **빨간색** 숫자 | 양수 (수익, 가격 상승) — 한국 주식 컨벤션 |
| **파란색** 숫자 | 음수 (손실, 가격 하락) |
| **검정색** 숫자 | 중립 (변동성, 스프레드 등 방향성 없는 수치) |

## 9. 결과 분석 — 가격 차트 읽기

**Lightweight Charts** 라인 차트가 전체 가격 추이를 보여줍니다.

```
가격
 ↑
 │    ╱╲      ╱╲      ╱╲
```

- 차트는 컨테이너 너비에 맞게 자동 리사이즈됩니다

## 10. 결과 분석 — 호가 깊이 시각화

### 화면 구성

호가 깊이 차트는 시뮬레이션 중 **50스텝마다 캡처된 호가창 스냅샷**을 보여줍니다.

```
┌─────────────────────────────────────┐
│ 호가 깊이           스냅샷 3 / 10   │
│ ◀════════════●═══════════════▶     │  ← 슬라이더
│                                     │
│   50,090  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓  250     │  ← 매도 5호가 (파란색)
│   50,080  ▓▓▓▓▓▓▓▓▓▓  180         │
│   50,070  ▓▓▓▓▓▓▓  130            │
│   50,060  ▓▓▓▓▓  100              │
│   50,050  ▓▓▓  60                  │  ← 매도 1호가
│   ─────── 구분선 ───────           │
│   50,040  ███  55                  │  ← 매수 1호가 (빨간색)
│   50,030  █████  90               │
│   50,020  ████████  150           │
│   50,010  ██████████  200         │
│   50,000  █████████████  300      │  ← 매수 5호가
└─────────────────────────────────────┘
```

### 읽는 법

- **위쪽 (파란색)**: 매도(ask) 호가 — 이 가격에 팔려는 수량
- **아래쪽 (빨간색)**: 매수(bid) 호가 — 이 가격에 사려는 수량
- **바 길이**: 해당 호가의 잔량 (최대 잔량 대비 비율)
- **구분선**: 매수 1호가와 매도 1호가 사이 = 스프레드

### 슬라이더 사용

상단 **슬라이더**를 좌우로 드래그하면 시간 순서대로 호가창 스냅샷이 변합니다.

- 시나리오 주입 전 호가와 주입 후 호가를 비교하세요
- **유동성 위기 시나리오**에서는 바가 급격히 짧아지는 것을 관찰할 수 있습니다
- **플래시 크래시**에서는 매수 호가가 순간적으로 사라지는 현상을 확인할 수 있습니다

## 14. 거버넌스 규칙 편집

거버넌스는 AI 에이전트의 행동을 **제한하는 안전장치**입니다.

### 14.1 현재 규칙 확인

기본 상태에서 규칙이 텍스트로 표시됩니다:

- 최대 주문 수량: 1,000
- 허용 액션: BUY, SELL
- 실거래 수동 승인: 예

### 14.2 규칙 편집

1. **"편집"** 버튼을 클릭합니다
2. 폼이 나타납니다:

| 항목 | 설명 | 기본값 | 의미 |
|------|------|--------|------|
| **최대 주문 수량** | 1건당 주문 가능한 최대 주식 수 | 1,000 | 초과 시 AI 주문 차단 |
| **허용 액션** | 허용되는 주문 유형 (콤마 구분) | BUY, SELL | 미등록 액션은 차단 |
| **실거래 수동 승인** | 실거래 모드에서 사람 확인 필요 여부 | 체크 | 체크 해제 = AI 자율 실거래 (위험!) |
| **거버넌스 활성화** | 규칙 전체 ON/OFF | 체크 | 해제 시 모든 AI 행동 무제한 (비권장) |

3. **"저장"** 클릭으로 즉시 반영됩니다
4. **"취소"** 클릭으로 변경을 버립니다

### 14.3 거버넌스 차단 흐름

```
[AI 에이전트] → execute_order(qty=5000)
                      │
                      ▼
              [GovernanceCheck]
              qty(5000) > max(1000)?
                      │ YES
                      ▼
              "차단: 수량 초과" → 감사 로그 기록
              AI에게 거부 응답 전달
```

## 15. 등록된 MCP 도구

MCP 서버에 등록된 도구 **5개**와 리소스 **1개**입니다.

### 도구 목록

| # | 도구명 | 기능 | 거버넌스 적용 |
|---|--------|------|-------------|
| 1 | **execute_order** | 주식 매수/매도 주문 실행 | O (수량 제한, 액션 검증, 실거래 차단) |
| 2 | **query_stock_data** | 캔들 데이터 조회 (1m~1M) | X (읽기 전용) |
| 3 | **fetch_sentiment** | 뉴스 감성 분석 데이터 조회 | X (읽기 전용) |
| 4 | **run_alpha_scan** | 알파 팩터 탐색 결과 조회 | X (읽기 전용) |
| 5 | **get_portfolio_status** | 보유 포지션 현황 조회 | X (읽기 전용) |

### 리소스

| 리소스 | URI 패턴 | 설명 |
|--------|---------|------|
| **실시간 호가** | `realtime_orderbook/{symbol}` | WebSocket 채널 정보 반환 |

### 각 도구 상세

#### execute_order

```
매개변수:
  - ticker: 종목코드 (6자리, 예: "005930")
  - action: BUY 또는 SELL
  - qty: 수량
  - price: 가격 (0 = 시장가)
  - mode: paper (모의) 또는 real (실거래)

거버넌스 검사 순서:
  1. qty > max_order_qty → 차단 "수량 초과"
  2. action ∉ allowed_actions → 차단 "액션 미허가"
  3. mode == "real" && 수동승인 필요 → 차단 "실전 모드는 사람 승인 필요"
```

#### query_stock_data

```
매개변수:
  - symbol: 종목코드 (예: "005930")
  - interval: 캔들 주기 (1m, 5m, 1h, 1d, 1w, 1M)
  - count: 조회 건수 (기본 200)

반환: 최근 10건의 캔들 데이터 (시가, 고가, 저가, 종가, 거래량)
```

#### fetch_sentiment

```
매개변수:
  - symbol: 종목코드
  - target_date: 날짜 (YYYY-MM-DD, 기본 오늘)

반환: 평균 감성 점수, 기사 수, 이벤트 스코어
```

#### run_alpha_scan

```
매개변수:
  - context: 시장 맥락 설명 (기본: "한국 주식시장")

반환: IC 기준 상위 10개 알파 팩터
```

#### get_portfolio_status

```
매개변수:
  - mode: paper 또는 real

반환: 보유 종목 목록 (종목코드, 수량, 평균단가)
```

### 로그 항목 구성

| 요소 | 설명 |
|------|------|
| **도구명** | 호출된 MCP 도구 (예: execute_order) |
| **상태 배지** | 성공(파랑) / 차단(보라) / 에러(빨강) |
| **차단 사유** | 차단된 경우 거버넌스 차단 이유 |
| **시각** | 호출 시각 |
| **실행 시간** | 처리 소요 시간 (ms) |

### 상태별 의미

| 상태 | 배지 색상 | 의미 |
|------|----------|------|
| **성공** | 파란색 | 도구 호출이 정상 실행됨 |
| **차단** | 보라색 | 거버넌스 규칙에 의해 실행이 거부됨 |
| **에러** | 빨간색 | 실행 중 오류 발생 (서버 에러, 네트워크 등) |

### 활용

- **차단 로그**가 많으면: 거버넌스 규칙이 AI 에이전트의 의도와 맞지 않을 수 있음 → 규칙 조정 검토
- **에러 로그**가 많으면: 외부 서비스(KIS API, DB) 연결 문제 확인 필요
- **실행 시간**이 길면: 네트워크 지연 또는 데이터 양 과다

## 17. 내부 구조 이해 — 가상 거래소

### 전체 아키텍처

```
[VirtualExchange] ─── 매 스텝 루프:
     │
     ├─ 1. 전체 에이전트의 decide() 호출
     │     각 에이전트가 현재 mid_price, 시장 상태를 보고
     │     {side, price, qty, type} 주문 리스트 반환
     │
     ├─ 2. 주문 → LimitOrderBook에 제출
     │     limit_order: 지정가 → 교차 시 즉시 체결, 잔량은 대기
     │     market_order: 시장가 → 호가 즉시 소화
     │
     ├─ 3. 체결(Fill) 결과 → 에이전트에 알림
     │     on_fill(): 현금/주식 업데이트, 평균단가 재계산
     │
     ├─ 4. 가격/거래량/스프레드 기록
     │     price_series[step] = mid_price
     │     volume_series[step] = 이번 스텝 총 체결량
     │     spread_series[step] = best_ask - best_bid
     │
     ├─ 5. (50스텝마다) 호가 깊이 스냅샷 저장
     │     depth_series.push({bids, asks})
     │
     ├─ 6. (지정 스텝) 시나리오 이벤트 주입
     │     inject_event(type, params)
     │
     └─ 7. 진행률 콜백 → WebSocket 브로드캐스트
           simulation:{run_id} 채널
```

### 초기 호가 시딩

시뮬레이션 시작 시, 빈 호가창에 **초기 유동성**을 주입합니다:

```
초기 가격: 50,000
틱 사이즈: 10

매수 호가 시딩: 50,000 → 49,990 → 49,980 → ... (5단계, 각 50~200주)
매도 호가 시딩: 50,010 → 50,020 → 50,030 → ... (5단계, 각 50~200주)

결과: 초기 스프레드 = 10 (1틱), 양측 유동성 확보
```

## 19. 내부 구조 이해 — 호가창 매칭

### LimitOrderBook 구조

```
                    매도 호가 (Ask)
                    ┌───────────────┐
          50,030   │  100주  (5번)  │  ← min-heap 최하단
          50,020   │  200주  (3번)  │
          50,010   │  150주  (1번)  │  ← best ask (매도 1호가)
                    ├═══════════════┤  ── 스프레드 ──
          50,000   │   80주  (2번)  │  ← best bid (매수 1호가)
          49,990   │  120주  (4번)  │
          49,980   │  250주  (6번)  │  ← max-heap 최상단
                    └───────────────┘
                    매수 호가 (Bid)

괄호 안 숫자 = 주문 도착 순서 (시간 우선)
```

### 매칭 규칙

1. **가격 우선**: 매수 주문은 높은 가격이 우선, 매도 주문은 낮은 가격이 우선
2. **시간 우선**: 같은 가격이면 먼저 접수된 주문이 우선
3. **교차 체결**: 매수 가격 ≥ 매도 가격이면 즉시 체결
4. **체결 가격**: 기존에 있던 주문(resting order)의 가격으로 체결

### 매칭 예시

```
기존 호가창: best_ask = 50,010 (150주), best_bid = 50,000 (80주)

→ 새 매수 지정가 주문: price=50,020, qty=200

매칭 과정:
  1차: ask 50,010 × 150주 체결 (체결가 50,010)
  2차: ask 50,020 × 50주 체결 (체결가 50,020)
  잔량: 없음 (200 - 150 - 50 = 0)

→ 새 시장가 매도 주문: qty=300

매칭 과정:
  1차: bid 50,000 × 80주 체결 (체결가 50,000)
  2차: bid 49,990 × 120주 체결 (체결가 49,990)
  3차: bid 49,980 × 100주 체결 (체결가 49,980)
  잔량: 없음 (300 - 80 - 120 - 100 = 0)
```

## 20. 응용: 실전 워크플로우

### 워크플로우 A: 전략 스트레스 테스트 (10분)

**목표:** 백테스트에서 성공한 전략이 극한 상황에서도 살아남는지 확인

1. **백테스트 페이지**에서 전략 실행 → 양호한 결과 확인
2. **시뮬레이션 페이지**로 이동
3. **flash_crash** 시나리오 선택 → "스트레스 테스트 시작"
4. 결과 확인: 전략 손익(PnL)이 -5% 이내인지?
5. **rate_shock** 시나리오로 재실행 → 결과 비교
6. **liquidity_crisis** 시나리오로 재실행 → 결과 비교
7. **supply_chain** 시나리오로 재실행 → 결과 비교

| 항목 | 안전 | 주의 | 위험 |
|------|------|------|------|
| **전략 PnL** | > -2% | -2% ~ -5% | < -5% |
| **MDD** | < 3% | 3% ~ 10% | > 10% |
| **충격 깊이** | < 1% | 1% ~ 5% | > 5% |
| **회복 시간** | < 50 스텝 | 50~200 | 미회복 (0) |

## 22. 트러블슈팅

### "스트레스 테스트 시작" 버튼이 반응하지 않음

**원인:** 이미 실행 중인 테스트가 있음
- 해결: 현재 실행이 완료될 때까지 대기하세요
- 또는: 서버를 재시작하면 RUNNING 상태가 FAILED로 전환됩니다

### 시뮬레이션이 FAILED로 끝남

**원인 1:** LLM 에이전트 API 키 미설정
- 확인: LLM 수가 1 이상인데 `ANTHROPIC_API_KEY`가 없음
- 해결: `.env`에 키 설정 또는 LLM 수를 0으로 설정

**원인 2:** 서버 메모리 부족
- 확인: total_steps가 10,000 이상이면서 에이전트가 300명 이상
- 해결: 총 스텝 또는 에이전트 수 감소

**원인 3:** 데이터베이스 연결 오류
- 확인: Docker 컨테이너(TimescaleDB)가 실행 중인지 확인
- 해결: `docker compose up -d`

### 진행률이 0%에서 멈춤

**원인:** WebSocket 연결 실패
- 확인: 브라우저 콘솔에서 `WebSocket connection failed` 에러
- 해결: 백엔드 서버가 실행 중인지 확인, VITE_WS_URL 환경변수 확인

실제 시뮬레이션은 서버에서 정상 실행 중일 수 있습니다. 2초마다 자동 폴링으로 결과가 표시됩니다.

### MCP 도구 목록이 "등록된 도구가 없습니다"로 표시

**원인:** fastmcp 패키지 미설치 또는 MCP 비활성
- 해결 1: `docker compose up -d --build` (Docker 이미지 리빌드)
- 해결 2: `.env`에 `MCP_ENABLED=true` 설정 후 재시작

### 모든 시나리오에서 충격이 너무 작음 (< 1%)

이것은 현재 시뮬레이션 엔진의 **알려진 제한사항**입니다.

**원인:** 에이전트 주문 수량이 호가창 잔량 대비 적음
- 해결: 향후 업데이트에서 에이전트 공격성 강화 예정
- 임시 조치: 노이즈 트레이더 수를 줄이면 (30~50) 유동성 감소 → 충격 증대

### 가격이 오래 정체(flatline)됨

**원인:** 에이전트 간 가격 수렴 후 교착 상태
- 해결: 총 스텝을 늘리면 교착이 해소되는 구간을 관찰할 수 있음
- 임시 조치: 다른 시드(seed) 값으로 재실행

### 차트 X축이 1970년 날짜로 표시

이것은 **알려진 UI 제한사항**입니다. 스텝 인덱스(0, 1, 2...)가 Unix 타임스탬프로 해석됩니다.
- 해결: 향후 업데이트에서 X축 포맷 변경 예정
- 차트 데이터 자체는 정상입니다 — X축 레이블만 무시하세요

## 23. 용어 사전

| 용어 | 영문 | 설명 |

| **SSE** | Server-Sent Events | 서버에서 클라이언트로 단방향 실시간 데이터 전송 |
| **스텝** | Step | 시뮬레이션의 시간 단위. 1스텝 = 전체 에이전트가 1회 의사결정 |
| **펀더멘탈** | Fundamental | 기업의 내재가치(재무, 성장성)에 기반한 투자 방식 |
| **차티스트** | Chartist | 가격 차트의 추세/패턴에 기반한 투자 방식 (기술적 분석) |
| **노이즈 트레이더** | Noise Trader | 정보 없이 무작위로 거래하는 참여자 (유동성 공급 역할) |
| **시장가** | Market Order | 현재 최우선 호가로 즉시 체결되는 주문 |
| **지정가** | Limit Order | 지정한 가격에만 체결되는 주문 (미체결 시 호가창에 대기) |
| **스프레드** | Spread | 매수 1호가와 매도 1호가의 차이. 유동성의 척도 |
| **틱 사이즈** | Tick Size | 호가 단위. 가격 변동의 최소 단위 |
| **MDD** | Maximum Drawdown | 최대 낙폭. 고점 대비 최대 하락폭 (%) |
| **충격 깊이** | Crash Depth | 시나리오 주입으로 인한 가장 깊은 가격 하락폭 |
| **회복 시간** | Recovery Steps | 충격 후 이전 고점의 99%까지 돌아오는 데 걸린 스텝 수 |
| **PnL** | Profit and Loss | 손익. 전략의 최종 수익 또는 손실 |
| **연환산 변동성** | Annualized Volatility | 일별 수익률 표준편차를 연간으로 환산한 값 |
| **거버넌스** | Governance | AI의 행동을 제한하는 규칙 체계 (안전장치) |
| **감사 로그** | Audit Log | 모든 도구 호출을 기록한 이력 (누가, 언제, 무엇을) |
| **손실 회피** | Loss Aversion | 동일 크기의 이익보다 손실을 더 크게 느끼는 심리 편향 |
| **군집 행동** | Herding | 다수의 행동 방향을 무비판적으로 따르는 심리 편향 |
| **시드** | Seed | 난수 생성기의 초기값. 같은 시드 = 같은 결과 (재현성) |
| **교차 체결** | Cross Match | 매수 가격 ≥ 매도 가격일 때 발생하는 즉시 체결 |
| **가격-시간 우선** | Price-Time Priority | 호가 매칭 규칙: 더 좋은 가격이 우선, 같은 가격이면 먼저 접수된 주문 우선 |
| **Lazy Cancellation** | — | 취소된 주문을 즉시 제거하지 않고, 힙 탐색 시 만나면 제거하는 기법 |
| **내재가치** | Intrinsic Value | 펀더멘탈 분석으로 산출한 주식의 "진정한 가치" |
| **SMA** | Simple Moving Average | 단순 이동평균 |
| **RSI** | Relative Strength Index | 상대강도지수 (0~100, 과매수/과매도 판단) |
| **카운터팩추얼** | Counterfactual | "만약 ~했다면?" 가설. 일어나지 않은 상황을 가정하고 시뮬레이션 |
